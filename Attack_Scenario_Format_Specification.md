# LoRaWAN Attack Scenario Format Specification (v1)## PurposeThis document defines a unified JSON scenario format for the LoRaWAN Offensive Security Testing Framework.The primary goals are:* standardize attack scenario definitions;* simplify scenario parsing and validation;* support future extensibility;* allow users to create custom scenarios independently;* ensure consistent syntax and structure across all attack types;* simplify CLI integration and automation.The format is designed for the MVP scope of the thesis project focused on offensive-security testing of LoRaWAN Network Servers via Semtech UDP packet-forwarder communication.---# Design PrinciplesThe scenario format must follow:* SOLID;* DRY;* KISS;* extensibility-first architecture;* schema-driven validation;* predictable naming conventions.Each scenario must use the same top-level structure regardless of attack type.Attack-specific parameters must only exist inside:```json"attack": {  "type": "...",  "config": {}}```This avoids uncontrolled schema growth and simplifies future attack additions.---# Naming Conventions## General Rules* Use `snake_case` for all fields.* Use explicit unit suffixes:  * `_sec`  * `_ms`  * `_hz`  * `_bytes`* Use lowercase enum values.* Hex values must not include `0x`.* Avoid ambiguous field names.---# Unified Scenario StructureAll scenarios must follow this structure:```json{  "schema_version": "1.0",  "scenario": {},  "target": {},  "gateway": {},  "device": {},  "attack": {},  "expected": {},  "logging": {}}```---# Top-Level Sections## schema_versionDefines scenario schema version.Example:```json"schema_version": "1.0"```---# scenarioContains metadata and classification.Example:```json"scenario": {  "id": "join-replay-basic",  "title": "Join Replay Basic",  "description": "Replay a JoinRequest using the same DevNonce.",  "category": "join_abuse",  "type": "join_replay",  "timeout_sec": 30}```## Required Fields| Field       | Description                || ----------- | -------------------------- || id          | Unique scenario identifier || title       | Human-readable title       || description | Short scenario description || category    | High-level attack family   || type        | Specific attack type       || timeout_sec | Maximum execution duration |---# targetDefines the target Network Server connection.Example:```json"target": {  "name": "chirpstack-local",  "transport": "semtech_udp",  "host": "127.0.0.1",  "port": 1700}```## Notes* Transport-specific configuration belongs here.* Do not place transport config inside gateway objects.---# gatewayDefines simulated gateway behavior.Example:```json"gateway": {  "gateway_eui": "0102030405060708",  "pull_data_interval_sec": 5,  "radio": {    "region": "EU868",    "frequency_hz": 868100000,    "data_rate": "SF7BW125",    "rssi": -60,    "snr": 7.5  }}```---# deviceDefines the simulated LoRaWAN device.Example:```json"device": {  "name": "attack-device-1",  "lorawan_version": "1.0.3",  "class": "A",  "activation": {    "mode": "OTAA",    "dev_eui": "0011223344556677",    "join_eui": "0011223344556677",    "app_key": "00112233445566770011223344556677"  }}```---# attackContains attack execution logic.All attack-specific parameters must exist only inside:```json"attack.config"```## Example```json"attack": {  "type": "join_replay",  "config": {    "replay_count": 3,    "delay_sec": 0.5,    "dev_nonce_strategy": "reuse_original",    "mic_strategy": "valid"  }}```---# expectedDefines expected secure behavior and validation criteria.This section is mandatory for offensive-security testing because the framework must validate security behavior instead of only generating traffic.Example:```json"expected": {  "secure_behavior": "replayed_join_requests_are_rejected",  "success_criteria": [    "first_join_request_is_accepted",    "replayed_join_requests_with_same_devnonce_are_rejected"  ]}```---# loggingDefines scenario logging behavior.Example:```json"logging": {  "level": "info",  "log_phy_payload": true,  "log_semtech_udp": true}```---# Allowed CategoriesThe following scenario categories are currently supported:```json[  "replay",  "join_abuse",  "mac_abuse",  "malformed_packet"]```---# Allowed Attack TypesThe following attack types are currently supported:```json[  "uplink_replay",  "join_replay",  "join_flood",  "mac_answer_abuse",  "malformed_mac_payload"]```---# Important Architectural Rules## Rule 1 — No Attack-Specific Top-Level BlocksThe following structure is prohibited:```json{  "replay": {},  "join_abuse": {},  "mac_command": {}}```This creates schema fragmentation and poor extensibility.Instead, all attack-specific logic must exist inside:```json"attack.config"```---## Rule 2 — Results Must Not Be Stored Alongside ScenariosScenario definitions and execution results must be separated.Recommended structure:```text/scenarios/results/logs```Examples:```text/scenarios/join-replay-basic.json/results/join-replay-basic.result.json/logs/join-replay-basic.log```---## Rule 3 — All Scenarios Must Be JSON Schema ValidatedThe framework must provide:```bashlorawan-sim validate scenario.json```Validation should include:* required fields;* enum validation;* type validation;* unit validation;* hex format validation;* transport compatibility validation.---# Replay Scenario RecommendationsReplay scenarios must explicitly define:* capture phase;* replay phase;* replay target;* FCnt strategy;* MIC strategy.Example:```json"attack": {  "type": "uplink_replay",  "config": {    "capture_phase": {      "perform_join": true,      "send_baseline_uplink": true,      "payload_hex": "01020304"    },    "replay_phase": {      "mode": "immediate",      "count": 3,      "delay_sec": 0.1    },    "fcnt_strategy": "reuse_original",    "mic_strategy": "reuse_original"  }}```---# Join Abuse RecommendationsJoin abuse scenarios should explicitly define:* DevNonce strategy;* DevEUI strategy;* MIC generation mode;* virtual device behavior.Example:```json"attack": {  "type": "join_flood",  "config": {    "request_count": 100,    "interval_sec": 0.05,    "virtual_devices": 10,    "dev_eui_strategy": "increment",    "dev_nonce_strategy": "random",    "mic_strategy": "valid"  }}```---# MAC Abuse RecommendationsThe MVP scope targets Network Server attacks through uplink traffic.Because of this:* uplink MAC command abuse is supported;* downlink-only commands must not be modeled as uplink attacks.Incorrect example:```json"command_type": "LinkADRReq"```because `LinkADRReq` is a Network Server → Device command.Preferred approach:```json"command_type": "LinkADRAns","malformation_type": "invalid_length"```or:```json"command_type": "RXParamSetupAns","fopts_length": 255```Focus on malformed uplink-side MAC behavior.---# Recommended CLI Workflow```bashlorawan-sim validate scenarios/join-replay-basic.jsonlorawan-sim run scenarios/join-replay-basic.json```---# Future ExtensibilityThe format must support future transports and integrations:* MQTT gateway bridges;* WebSocket packet-forwarders;* proprietary vendor integrations;* multi-gateway scenarios;* roaming simulations;* distributed attack orchestration.The current schema must remain backward-compatible whenever possible.---# Final RecommendationThe scenario format should behave as a stable public API for attack definitions.Future development must prioritize:* backward compatibility;* predictable schema evolution;* strong validation;* modular attack configuration;* user-defined scenario extensibility.
